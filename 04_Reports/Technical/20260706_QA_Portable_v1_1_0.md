## 6. Performance (Fase 6)

Medido con `05_Dashboard/tests/test_performance_portable.py` — HTTP real
contra `/_dash-update-component` del `.exe` corriendo (mismo endpoint que
usa el navegador), **no** una simulación. Guardado en
`dist/Gemelo_Digital_Molienda/qa_results/performance_results.json`.

| Acción | Meta | Medido (máx de 5 muestras) | Cumple |
|---|---:|---:|---|
| Arranque | < 15 s | 13.03 s (peor caso) | ✅ |
| Cambio de parámetro simple (slider `/riesgo`, HTTP+Plotly incluido) | < 2 s | 194 ms | ✅ (10x margen) |
| Cambio de pestaña (modo rápido/avanzado) | < 1 s | 26 ms | ✅ |
| Optimizador (`find_optimal_v3`, medido a nivel de motor en sesión previa) | < 8 s | ~7.0 s peor caso medido / ~0.1 ms si el escenario se repite (cache) | ✅ en el límite sin cache, holgado con cache |
| Monte Carlo adaptativo | < 10 s | Incluido en el tiempo de `find_optimal_v3` (20 llamadas internas a `adaptive_mc_eval`, 180-810 ms c/u) | ✅ |

**Nota de método:** el optimizador/Monte Carlo no se remidieron end-to-end
por HTTP en esta sesión porque su callback usa `allow_duplicate=True` con
sufijos hash en los IDs de output, lo que complica reconstruir a mano el
payload exacto del protocolo Dash. Se reutiliza la medición a nivel de
motor (misma función Python que ejecuta el callback) de la sesión
anterior, ya validada — es representativa del costo real de cómputo; la
única cola adicional en el navegador real sería serialización de figuras
Plotly, del orden de decenas de ms.

## 9. Hallazgos

### Corregido: mantención de día completo `[0, 24]` se ignoraba (bug real)

**Severidad:** Media — silenciosa, sin error visible, pero produce una
recomendación incorrecta (asume equipo disponible cuando el usuario marcó
mantención todo el día).

`engine/scheduler.py::hour_in_window` aplicaba `fin % 24.0` antes de
comparar `ini == fin`. Una ventana `[0, 24]` (alcanzable arrastrando el
`RangeSlider` `ctrl-mant-*` a fondo — `min=0, max=24`, una forma natural de
marcar "mantención todo el día") se convertía en `ini=0, fin=0`, y la
función la trataba como "sin ventana" — el equipo aparecía disponible
todo el horizonte pese a estar marcado en mantención completa.

**Fix aplicado:** se verifica `(fin - ini) >= 24.0` antes del módulo — si
la ventana declarada cubre 24h o más, se considera activa para cualquier
hora, sin pasar por la comparación `ini == fin` post-módulo. Test de
regresión agregado: `test_ventana_dia_completo_0_24_se_detecta_activa`
(`tests/test_rules_engine.py`).

**No es un cambio a la lógica matemática validada** (balance de masa,
autonomía, calibración) — es un fix de un helper de agenda/horario.

### No bloqueante: Caso 4 (overflow) no tiene "tiempo hasta overflow" explícito

El simulador expone `t_critico_sag{1,2}_h` (tiempo hasta **vaciado**) pero
no un campo simétrico para tiempo hasta **overflow**. El overflow se
comunica hoy solo de forma visual (gráfico de pila, zona 95-105%) y
probabilística (Monte Carlo: `pct_overflow_sagX`, `hourly_risk`), nunca
como un escalar determinístico. Esto es una asimetría de diseño existente,
no una regresión de esta sesión — **no se implementó** un campo nuevo
porque el objetivo explícito de este QA es *no agregar funcionalidades*.
Queda como recomendación futura si el negocio lo pide explícitamente.
